## 8.1 The Reference Diagram

![Role hierarchy, GDPR inheritance and live-chat flow](/product/assets/role-hierarchy-and-live-chat-flow.png)

The diagram captures three things in one frame:

1. **Top-right:** the role hierarchy (Platform Admin → Tenant Admin → Counselor Admin → Counselor) with the **MFA must** annotation on Platform Admin and Tenant Admin, and the **Imprint / Data Policy / GDPR Agreement** documents attached to each tier.
2. **Middle:** how legal text inherits down the tree (solid arrows = create, dashed = inherit / fallback) into the **Counseling Centers** and from there to the live-chat ticket flow.
3. **Bottom-left:** two iPhone mockups showing the **Waiting Room (Warteraum)** with a GDPR consent pop-up ("Bestätigen") and the queue position indicator ("23 Noch vor Ihnen").

We document each visible screen below, in the order a real client or staff member encounters them.

## 8.2 Screens

### 8.2.1 Public Live-Chat Landing → Waiting Room (Client, mobile)

> The bottom-left phone in the diagram. German label: *Warteraum*.

| Aspect | Detail |
|---|---|
| **Screen name** | Waiting Room — Pre-consent state |
| **Audience** | Anonymous Client, mobile-first |
| **Purpose** | Capture initial GDPR consent and put the client into the queue. |
| **Key components** | Header *Warteraum* with timer; pseudonym block (*"DEIN PSEUDONYM — geschmeidiges Kanninchen Kim"*); explanatory text ("Hallo, bevor wir fortfahren, bestätigen Sie bitte für diese Sitzung unserer auflagig diagnostizen Benutzerinnen…"); animated cat avatar; bottom modal with "Bestätigen" CTA and language / app-grid icons. |
| **User actions** | Read consent text → tap **Bestätigen** to confirm the platform-default GDPR. (Optional) tap the language globe to switch language. |
| **Backend interaction** | On entry: `POST /live-chat/anonymous` returns ticket id + cookie. On confirm: `POST /live-chat/consent?level=platform` records consent. Frontend then transitions to the queued state. |
| **Notes / rules** | Pseudonym is deterministic-on-randomness; never reused. The page must default to the **browser's language** (English fallback, *not* German) on first paint. |

### 8.2.2 Waiting Room — Queued state (Client, mobile)

> The right-hand phone in the diagram.

| Aspect | Detail |
|---|---|
| **Screen name** | Waiting Room — Queued |
| **Audience** | Anonymous Client |
| **Purpose** | Reassure the client during the wait, surface E2EE guarantee, allow self-service routing changes. |
| **Key components** | Privacy reassurance banner (*"Vielen Dank, übrigens zum Schutz Ihrer Privatsphäre sind alle Nachrichten Ende-zu-Ende verschlüsselt und werden noch nach 48 Stunden automatisch gelöscht."*); emoji breathing-game button; the **"23 Noch vor Ihnen"** queue counter; an "Vor Ort Beratungsanfrage absenden" CTA (file an in-person enquiry instead); a *Spiel zu warten* (game while you wait) link. |

### 8.2.3 GDPR Consent #2 (Client, mobile, full-screen)

> Implied by the central diagram node *"Client must agree to GDPR of Counseling Center"*.

| Aspect | Detail |
|---|---|
| **Screen name** | Counseling-Center GDPR Consent |
| **Audience** | Anonymous Client at the moment a counselor accepts their ticket |
| **Purpose** | Take the **counseling-center-specific** GDPR agreement before opening the chat room. |
| **Key components** | Full-screen sheet; centered "Einverstanden" / "Bestätigen" CTA; a "Reject" link; the counseling-center's name and Imprint excerpt. |
| **User actions** | Read; **Confirm** (chat opens) or **Reject** (session aborted, pseudonym wiped). |

## 8.3 Cross-Screen Interaction Patterns

### 8.3.1 The two GDPR popups are different

A frequent confusion: the popup in the Waiting Room is **platform-level**; the popup that appears at counselor-pickup is **center-level**. Visually they should be **distinguishable** (different headers, different "powered by …" footers).

### 8.3.2 The deactivated state must be informative

A live-chat section toggled off shows:
- Greyed-out ticket list.
- A banner *"Live chat is currently off. Activate it to start receiving requests."* with a CTA pointing to the activation toggle.
- The same count of historical tickets (read-only) for context.

### 8.3.3 The "old" inquiry section stays in place

The legacy enquiry chat (zip-code based, registered) lives **alongside** live chat on the counselor's dashboard — they share the navigation but are separate sections.

### 8.3.4 Mobile-first

The Waiting Room and the Live Chat are designed mobile-first (the Figma mockups are iPhone-15-Pro). The admin panel is desktop-first (data tables, multi-column).

## 8.4 Open Design Tickets (From Huddle)

- **Live-chat onboarding redesign** — the breathing-game / waiting-room flow has half-built screens; needs to be finalised. *(Ticket: "fix live chat onboarding screen design".)*
- **Live-chat link generation UI** — overengineered today, simplify to "one button per topic per agency".
- **Bilingual UI** — all screens must support EN/DE side-by-side, default to browser language with EN fallback.
- **Design-system alignment** — admin panel screens have visible drift from Figma; refactoring backlog item.
- **Removed sections must be visibly deactivated, not hidden** — global rule.

## 8.7 Animation & Micro-interaction Rules (Adopted)

From the Figma annotations, ORISO globally adopts:

- Animations play **once** per click; never loop.
- Animation playback speed **≤ 0.7×**.
- Empty states use the same neutral micro-illustration set across the app.
- Hover state is always present (never "hover-only" interactions, but always a visual hint).
- Button container "remembers" its last open/closed state per user.
